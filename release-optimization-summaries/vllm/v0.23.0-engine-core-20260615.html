vLLM v0.23.0 Engine Core 优化总结

整理自 vLLM 2026-06-15 发布帖和 v0.23.0 release notes，重点覆盖 DeepSeek-V4、Model Runner V2、统一 Parser 和 multi-tier KV cache offloading。

DeepSeek-V4 Model Runner V2 Unified Parser KV Offloading
一句话概括：这版不是单个 kernel 的点状优化，而是围绕 DSv4 长上下文/MoE、MRv2 执行器默认化、OpenAI API parser 统一、KV 分层 offload 四条线，把 vLLM 的高性能路径和服务扩展路径继续产品化。

按主线理解

DeepSeek-V4

重点是 sparse MLA、Mega-MoE、sliding-window KV cache 和 compile 依赖剥离。它让 DSv4 从 day-0 支持进入更稳定、更高性能的专用路径。

Model Runner V2

MRv2 继续扩大默认覆盖面，并补齐 sampler、CUDA graph、pipeline parallel 和 Gemma 4 MTP。这是执行器默认化的信号。

Unified Parser

统一 reasoning 与 tool-call parsing 后，OpenAI Chat / Responses 等入口可以共享 parser 逻辑，减少模型特化 parser 的分叉。

KV Offloading

从单层 offload 走向多层 KV cache 管理：object store、HMA、tiering 和 per-request policy 组合起来，面向长上下文和 disaggregated serving。

后续跟踪建议

  • 对 DSv4：跟踪 TRTLLM-gen attention kernel 与 FlashInfer/FlashMLA 路径在长上下文 decode/prefill 上的实际收益差异。
  • 对 MRv2：关注 Llama/Mistral 默认启用后的回归问题，尤其是 speculative decoding、CUDA graph 和 pipeline parallel 组合场景。
  • 对 KV offloading：重点看 object-store secondary tier 与 HMA tiering 的真实延迟尾部，以及 per-request policy 是否会进入生产服务配置。
  • 对 Parser：关注新模型 tool parser 是否逐步迁移到统一接口，避免 Responses API 和 Chat API 表现不一致。
