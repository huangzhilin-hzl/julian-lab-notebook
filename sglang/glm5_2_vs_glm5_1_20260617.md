# GLM-5.2 vs GLM-5.1 技术对比

记录时间：2026-06-17

结论口径：基于官方 Hugging Face 配置、模型卡、vLLM recipe，以及 IndexCache 论文信息整理。

## 结论摘要

GLM-5.2 不是简单参数规模升级。它和 GLM-5.1 的主干尺寸基本一致，核心变化集中在四个方向：

| 方向 | GLM-5.1 | GLM-5.2 | 影响 |
| --- | --- | --- | --- |
| 上下文长度 | 202,752 tokens | 1,048,576 tokens | 5.2 支持约 1M 上下文，适合长仓库、长文档、长 agent 轨迹。 |
| Sparse attention indexer | DSA indexer 每层独立计算，配置里没有显式共享模式 | IndexShare：`full/shared` 跨层复用 indexer | 主要降低长上下文下 DSA indexer 的 FLOPs 和 indexer cache。 |
| 思考强度控制 | chat template 没有多档 effort 注入 | `reasoning_effort=high/max`，默认 `max` | 通过模板/API 调节 reasoning token 数，在性能和延迟之间取舍。 |
| Speculative decoding | 有 `num_nextn_predict_layers=1` | 有 MTP 层，并在 vLLM recipe 中明确启用 `num_speculative_tokens=5` | 5.2 更强调 MTP speculative decoding 的工程化部署路径。 |

一句话判断：**GLM-5.1 已经是 DSA + MoE + MTP 的强 coding/agent 模型；GLM-5.2 的主要增量是把它推向 1M 上下文和更可部署的长上下文 coding agent。**

## 架构配置对比

| 配置项 | GLM-5.1 | GLM-5.2 |
| --- | ---: | ---: |
| `model_type` | `glm_moe_dsa` | `glm_moe_dsa` |
| `num_hidden_layers` | 78 | 78 |
| `hidden_size` | 6144 | 6144 |
| `num_attention_heads` | 64 | 64 |
| `num_key_value_heads` | 64 | 64 |
| `kv_lora_rank` | 512 | 512 |
| `qk_rope_head_dim` | 64 | 64 |
| `index_topk` | 2048 | 2048 |
| `num_nextn_predict_layers` | 1 | 1 |
| `max_position_embeddings` | 202,752 | 1,048,576 |
| `index_topk_freq` | 未显式设置 | 4 |
| `index_skip_topk_offset` | 未显式设置 | 3 |
| `index_share_for_mtp_iteration` | 未显式设置 | `true` |

GLM-5.2 的关键配置片段：

```json
{
  "max_position_embeddings": 1048576,
  "index_topk": 2048,
  "index_topk_freq": 4,
  "index_skip_topk_offset": 3,
  "index_share_for_mtp_iteration": true,
  "num_nextn_predict_layers": 1
}
```

`indexer_types` 的前几层模式如下：

```text
full, full, full, shared, shared, shared, full, shared, shared, shared, full, ...
```

解释：

| 模式 | 含义 |
| --- | --- |
| `full` | 当前层运行 DSA indexer，重新为 query 选择 top-k 历史 token。 |
| `shared` | 当前层不重新运行 indexer，直接复用前一个 `full` 层的 top-k indices。 |

这对应发布文案里的 **IndexShare**：每约 4 个 sparse attention layers 共享一次 indexer。

## IndexShare / IndexCache

GLM-5.2 发布文案叫 **IndexShare**，论文标题叫 **IndexCache: Accelerating Sparse Attention via Cross-Layer Index Reuse**。两者描述的是同一类机制：DSA sparse attention 中，多个相邻层复用同一份 top-k token selection。

普通 DSA 的执行方式：

| 步骤 | 说明 |
| --- | --- |
| 1 | 每层用轻量 indexer 对历史 token 打分。 |
| 2 | 选择 top-k token，例如 GLM-5.2 中 `index_topk=2048`。 |
| 3 | 主 attention 只关注这 2048 个 token，从 dense attention 降为 sparse attention。 |

IndexShare 后的执行方式：

| 层类型 | 行为 |
| --- | --- |
| Full layer | 正常运行 indexer，生成 top-k indices。 |
| Shared layer | 跳过 indexer，复用最近 full layer 的 top-k indices。 |

收益来自两个地方：

| 收益点 | 说明 |
| --- | --- |
| 降低 indexer FLOPs | 1M context 下，indexer 本身扫描长上下文，成本很高。减少 3/4 indexer 调用会明显降 prefill/long decode 成本。 |
| 降低 indexer cache | 只需要给 full indexer 层保存 indexer key/state，而不是每层都保存。 |

需要注意：**IndexShare 主要优化 DSA indexer，不直接消除主 KV cache 随 context length 线性增长的问题。**

## Flexible Effort

GLM-5.2 的 `chat_template.jinja` 增加了 reasoning effort 注入：

{% raw %}
```jinja
{%- set effective_reasoning_effort = 'high' if reasoning_effort is defined and reasoning_effort == 'high' else 'max' -%}
{%- if (enable_thinking is not defined or enable_thinking) and effective_reasoning_effort is not none -%}
<|system|>Reasoning Effort: {{ effective_reasoning_effort | capitalize }}
{%- endif -%}
```
{% endraw %}

实际含义：

| 模式 | 解释 | 适合场景 |
| --- | --- | --- |
| `high` | 控制 reasoning token 数，降低延迟和成本 | 常规 coding、较明确的修复、短链路任务 |
| `max` | 默认更深 thinking | 难题、复杂 repo、长链路 agent task |
| `enable_thinking=false` | 模板会关闭 thinking 输出 | 低延迟、直接回答、非复杂推理任务 |

这不是结构性 FLOPs 开关，而是通过 post-training + prompt template 让模型按不同 effort 生成不同长度/深度的 reasoning。

## MTP / Speculative Decoding

两者配置中都有：

```json
{
  "num_nextn_predict_layers": 1
}
```

GLM-5.2 的权重中存在额外的 `model.layers.78.*`，用于 next-token / multi-token prediction 相关路径；标准 Transformers `GlmMoeDsaForCausalLM` 的普通 `generate()` 路径不直接体现完整 speculative serving 逻辑，真正部署时主要依赖 vLLM/SGLang 这类 serving 框架。

vLLM GLM-5.2 recipe 推荐：

```bash
vllm serve zai-org/GLM-5.2-FP8 \
  --kv-cache-dtype fp8_e4m3 \
  --tensor-parallel-size 8 \
  --speculative-config.method mtp \
  --speculative-config.num_speculative_tokens 5
```

解释：

| 参数 | 含义 |
| --- | --- |
| `--speculative-config.method mtp` | 使用模型内置 MTP 层做 draft。 |
| `--speculative-config.num_speculative_tokens 5` | 每轮最多 draft 5 个 token，再由主模型验证。 |
| acceptance length | 一轮 speculative decoding 中平均被接受的连续 draft token 数。 |

发布文案中 “acceptance length up to 20%” 表示 5.2 的 MTP draft 更容易被主模型接受，decode loop 可减少主模型验证轮数。它主要优化 decode latency，不显著减少 1M prompt 的 KV cache。

## KV Cache 估算

下面只估算 KV cache payload，不包括模型权重、activation、allocator、paged-block metadata、fragmentation、CUDA graph workspace 等工程开销。

### 公式

普通 MHA/GQA 展开 KV：

```text
tokens * layers * num_key_value_heads * (Dk + Dv) * bytes_per_element
```

GLM-5.x 的 MLA 主 KV：

```text
tokens * layers * (kv_lora_rank + qk_rope_head_dim) * bytes_per_element
```

GLM-5.2 的 DSA indexer cache，按 IndexShare 后 full indexer 层估算：

```text
tokens * full_indexer_layers * index_head_dim * bytes_per_element
```

### 1M context 下 GLM-5.2 的 cache 规模

假设 batch=1、sequence=1、context=1,048,576：

| 口径 | 结果 |
| --- | ---: |
| 如果按普通 MHA/GQA 展开 KV，BF16 | 4.875 TiB |
| GLM-5.2 MLA 主 KV，BF16 | 87.75 GiB |
| GLM-5.2 MLA 主 KV，FP8 | 43.875 GiB |
| DSA indexer cache，IndexShare BF16，约 21 个 full indexer 层 | 5.25 GiB |
| GLM-5.2 合计，BF16 主 KV + BF16 indexer | 93.0 GiB |
| GLM-5.2 合计，FP8 主 KV + BF16 indexer | 49.125 GiB |

和 GLM-5.1 粗略对比：

| 模型 | Context | MLA 主 KV BF16 | Indexer cache BF16 | 合计估算 |
| --- | ---: | ---: | ---: | ---: |
| GLM-5.1 | 202,752 | 16.97 GiB | 3.77 GiB，按无共享估算 | 20.74 GiB |
| GLM-5.2 | 1,048,576 | 87.75 GiB | 5.25 GiB，按 IndexShare 估算 | 93.0 GiB |

判断：

| 问题 | 结论 |
| --- | --- |
| 1M context 是否导致 KV cache 大幅增长？ | 是，主 KV 仍随 token 数线性增长，5.2 相比 5.1 的 context 长度是约 5.17 倍。 |
| 为什么没有膨胀到 TiB 级别？ | MLA 只缓存 compressed latent KV + RoPE key，不缓存展开后的每头 K/V。 |
| IndexShare 主要省什么？ | 省 DSA indexer 的计算和 indexer cache，不是主 KV。 |
| FP8 KV 的作用？ | 直接把 MLA 主 KV payload 从 87.75 GiB 降到 43.875 GiB。 |

## 推理部署优化手段

| 优化手段 | 作用 | 对 GLM-5.2 的意义 |
| --- | --- | --- |
| MLA compressed KV | 缩小 KV cache 维度 | 从普通 KV 的 TiB 级降到百 GiB 内。 |
| FP8 KV cache | 降低 KV payload 字节数 | 1M context 下从 87.75 GiB 主 KV 降到 43.875 GiB。 |
| IndexShare / IndexCache | 减少 DSA indexer 计算和 indexer cache | 长上下文 prefill 和 long decode 的关键优化。 |
| Paged KV | 降低连续显存分配压力，提升复用和调度效率 | 长上下文 serving 必需。 |
| Chunked prefill | 把超长 prompt 分块处理 | 降低 prefill 峰值压力，便于调度。 |
| Prefix caching | 复用共享 prompt 前缀 | 多请求共享长上下文时收益明显。 |
| Context parallel / KV sharding | 把长上下文 KV 分摊到多卡 | 1M context + batch 并发时很关键。 |
| KV offload / tiered cache | 把冷 KV 放到 CPU/NVMe | 换取容量，牺牲延迟。 |
| Sliding window / eviction / RAG | 减少需要保留的全量 token | 质量换内存，不等价于严格 1M 全量回看。 |
| MTP speculative decoding | 降低 decode latency | 主要改善输出阶段，不解决 prompt KV 大小。 |

## 能力侧变化

官方 benchmark 中，GLM-5.2 相比 GLM-5.1 提升较明显的是 coding、terminal、agent、long-context 类任务：

| Benchmark | GLM-5.1 | GLM-5.2 | 变化 |
| --- | ---: | ---: | ---: |
| SWE-bench Pro | 58.4 | 62.1 | +3.7 |
| NL2Repo | 42.7 | 48.9 | +6.2 |
| DeepSWE | 18.0 | 46.2 | +28.2 |
| ProgramBench | 50.9 | 63.7 | +12.8 |
| Terminal Bench 2.1 | 63.5 | 81.0 | +17.5 |
| MCP-Atlas | 71.8 | 76.8 | +5.0 |
| Tool-Decathlon | 40.7 | 48.2 | +7.5 |
| GPQA-Diamond | 86.2 | 91.2 | +5.0 |

这些提升和上面的技术路线是匹配的：coding agent 场景通常需要更长上下文、更长工具轨迹、更强 reasoning effort 控制，以及更低 decode latency。

## 采用建议

| 场景 | 更适合的模型 | 原因 |
| --- | --- | --- |
| 普通 200K 内 coding / agent | GLM-5.1 或 GLM-5.2 | 5.1 已经足够强；5.2 成本更高但能力更好。 |
| 1M context、长仓库、长日志、长文档 | GLM-5.2 | 5.2 明确支持 1M context，并有 IndexShare 降低长上下文 indexer 成本。 |
| 需要低延迟 thinking 控制 | GLM-5.2 | 支持 `reasoning_effort=high/max` 和 `enable_thinking=false`。 |
| 需要 speculative decoding 部署 | GLM-5.2 | 官方 vLLM recipe 明确给出 MTP 5 draft token 路径。 |
| 显存非常紧张 | 先评估 GLM-5.1 或 GLM-5.2 FP8 KV | 1M context 下 GLM-5.2 单请求 KV payload 也可能接近 50 GiB 级别。 |
