2. 收集 ncu profile

# M=64
ncu --set full -k regex:humming \
    --launch-skip 50 --launch-count 1 \
    -o ncu_m64_fp8fp4 \
    python benchmarks/bench_humming.py \
        --shape_n 4096 --shape_k 4096 \
        --a_dtype float8e4m3 --b_dtype float4e2m1 --bs_dtype float8e8m0 \
        --c_dtype bfloat16 \
        --weight_scale_group_size 32 --input_scale_group_size 0 \
        --shape_m_list 64

# M=4096（命令同上，--shape_m_list 4096，输出 ncu_m4096_fp8fp4.ncu-rep）
注意：ncu --set full 会重放 kernel ~39 次采集各种 counter；bench 表格里看到的 time 会被严重放大（M=64 显示 0.19 ms），那不是真实 kernel 时间，要看 .ncu-rep 里的 Duration 指标。

5. 解读

5.1 M=4096：compute-bound 接近极限 ✅

Speed Of Light 自动诊断（ncu Rule 文本）：

This workload is utilizing greater than 80.0% of the available compute
or memory performance of the device.

证据链：

  • SM Throughput 88.82% —— GPU 整体很忙
  • ALU pipe 占 45.20% —— 最忙的不是 Tensor Core，是 ALU（dequant 的位运算、地址计算）
  • Tensor pipe 22.21% —— TC 是被"按需喂"的
  • DRAM Throughput 1.87% —— 几乎不从 DRAM 读，全在 L1/L2 里转
  • L1 hit 96.77%, L2 hit 95.25% —— 缓存命中率极高
  • compute_tops = 253.43 / 296 ≈ 86% of FP8 peak

Top stall 是 long_scoreboard (51%) —— 大量周期在等数据（smem load）。但这不是病：WGMMA 是异步的，发完就能算别的，长 stall 被指令流后续藏住，所以 SM throughput 仍能到 88%。

Pipeline 是否充分使用？答案：✅ 是的，~86% 接近峰值。

剩下的 14% gap 推测来自：

  • WGMMA 异步发射但仍要 wait_group 收尾（wait 占 8.2%）
  • mbarrier 偶有等待（barrier 30.7%，但被 long_scoreboard 摊掉了，因为两边经常并行）
  • ALU 跟 Tensor Core 抢 issue 槽（dequant 比较重）
  • 14.35% 的 shared store 有 bank conflict（ncu 提示，影响很小）

5.2 M=64：被 problem-size + barrier 双杀 ❌

Speed Of Light 自动诊断：

This workload exhibits low compute throughput and memory bandwidth
utilization relative to the peak performance of this device. Achieved
compute throughput and/or memory bandwidth below 60.0% of peak typically
indicate latency issues.

证据链：

  • SM Throughput 39.76% —— 一半多的周期 GPU 没事做
  • Tensor pipe 9.94% —— TC 几乎没启动起来
  • DRAM Throughput 11.48% —— 不是 memory-bound
  • L1 hit 74.5%, L2 hit 49% —— cache 是凉的，kernel 太短没机会暖
  • compute_tops = 45.71 / 296 ≈ 15% of FP8 peak

Top stall 是 barrier (36%)，紧跟 long_scoreboard (31%) —— mbarrier 等待主导。每发出 1 条指令，warp 平均花 2.24 cycles 在 mbarrier 上等。

为什么 mbarrier 这么贵？因为：

  1. 每 SM 只跑 1 个 block（寄存器限制），没有 block-to-block 流水可言。
  2. 单 block 的 K-loop 太短：block_k=128，K=4096，每 SM 通过 Stream-K 只分到 4096 / 78 ≈ 52 列的 K 工作，不到 1 个完整 K-tile 的工作量。
  3. Pipeline 还没进入稳态就要 drain，producer 和 consumer 互相"踩着对方的脚"等待。

Pipeline 是否充分使用？答案：❌ 远没有。

但根因不是 humming 的 pipeline 设计本身有问题，而是：

  1. Problem size 太小，每个 block 工作量小于 pipeline depth
  2. auto-tuner 选的 tile (64, 128, 128) 对 M=64 不够极致
  3. kernel launch / drain 的固定成本对 20 μs 的 kernel 占比过大

6. 综合结论 + 优化方向

6.1 「pipeline 充分使用」的判定

Workload判定主要瓶颈优化空间
M=4096 ✅ 充分使用 长延迟内存等待（合理） 小（5~10%，bank conflict、ALU/Tensor 平衡）
M=64 ❌ 严重浪费 mbarrier + 短 K-loop + 寄存器锁死 occupancy 大（理论 3-5×，但需要 kernel 设计变更）

6.2 可行的 M=64 优化方向

改 tuning_config（轻量，先试这些）

  • 试更小的 block_n（64 而不是 128）→ 让单 block 工作量更短但 grid 更密
  • 试更小的 block_k（64 而不是 128）→ K-loop 多几轮，pipeline 能多周转
  • 降 num_stages（3 而不是更多）→ pipeline 浅了反而对短 kernel 更好

改 kernel 设计（重量）

  • 加 persistent kernel + multi-block scheduling：让一个 launch 处理多个 problem，摊销 launch / drain cost
  • 加 dual consumer warpgroup：把 epilogue 和下一个 mainloop 重叠

改业务（最实用）

  • Decode 阶段（小 M）本来就是 memory-latency-bound，与其压 GEMM 不如：
    • batch 多个 request 凑大 M
    • 用专门的 GEMV kernel（针对 M=1~16 优化）替代 GEMM

6.3 对比的更广含义

现象M=64M=4096
Tensor Core active9.94%22.21%
ALU active25.99%45.20%
ALU / Tensor 比2.6×2.0×

两个 size ALU 都比 Tensor 高 2-3 倍，说明 MXFP4 dequant + 地址计算 + 标量算术在 humming 里非 trivial。如果想进一步压榨 H20 FP8 峰值，可能要：

  • 把 dequant 完全移出 mainloop（pre-process pass）
  • 用 SASS 级整数 LUT 替代浮点 dequant

但这是 86% → 95% 的最后一公里优化，对 M=4096 而言投入产出比一般。

8. ncu 使用上的踩坑笔记

  1. --print-summary 默认走 "Rule"（自动诊断）视图，输出的是 ncu 自己的建议文本，不是原始数值。要拿数值用 --page raw --csv。
  2. bench 在 ncu 包裹下显示的 time 不可信：ncu 重放 kernel ~39 次，triton.testing.do_bench 测到的是一个被放大的值。真实时间看 ncu 报告里的 gpu__time_duration.sum 或 SOL section 的 Duration。
  3. --set full 缓存的指标集是固定的。如果想查的指标不在缓存里（比如 sm__pipe_tensor_op_qmma_*），需要重新跑 ncu profile，用 --metrics ... 显式指定，不能事后从 .ncu-rep 补。
  4. H20 没有 NVLink C2C / cluster 接口，所以 ctc__rx_bytes_* / cluster__* 等指标采集会报 warning，正常忽略。
  5. ncu 需要 GPU performance counter 权限。pod 里通常已经放开；本地机器报 ERR_NVGPUCTRPERM 时要 sudo 或修 nvidia 驱动参数。
  6. CSV 字段顺序按指标名字母序输出，不是按 --metrics 传入的顺序 —— 解析时要按列名匹配。
